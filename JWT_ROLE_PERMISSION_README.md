# JWT Role and Permission Guide

This project uses JWT authentication together with role-based permission mapping.

The flow is:

1. A user logs in through `POST /api/v1/auth/login`.
2. The server validates the credentials from environment variables.
3. The server resolves the user's roles into permissions.
4. The server signs a JWT containing `sub`, `roles`, and `permissions`.
5. Protected routes use middleware to:
   - verify the token
   - attach the decoded payload to `req.user`
   - check whether the user has the required role or permission

## How Login Works

Login is handled in `src/services/auth.service.js`.

The service accepts either:

```json
{
  "username": "admin",
  "password": "password123"
}
```

or:

```json
{
  "email": "admin",
  "password": "password123"
}
```

The login service:

1. Reads the expected username from `AUTH_USERNAME`.
2. Reads the password from `AUTH_PASSWORD_HASH` or `AUTH_PASSWORD`.
3. Validates the submitted password with `bcrypt`.
4. Reads the user's roles from `AUTH_ROLES`.
5. Resolves permissions from those roles.
6. Generates a JWT and returns it.

If `AUTH_ROLES` is not set, the default role is `admin`.

## Environment Variables

These environment variables control auth:

- `AUTH_USERNAME`
- `AUTH_PASSWORD`
- `AUTH_PASSWORD_HASH`
- `AUTH_SECRET`
- `AUTH_TOKEN_TTL_SECONDS`
- `AUTH_ROLES`

Current development defaults are:

- `AUTH_USERNAME=admin`
- `AUTH_PASSWORD=password123`
- `AUTH_SECRET=dev-secret-change-me`
- `AUTH_TOKEN_TTL_SECONDS=3600`
- `AUTH_ROLES=admin`

## JWT Payload Shape

The token is created in `src/utils/token.js` using `HS256`.

Important security note:

- this JWT is signed, not encrypted
- anyone holding the token can decode its payload
- the signature is what prevents tampering
- sensitive secrets should not be stored inside the JWT payload

The payload generated by `AuthService.login()` looks like this:

```json
{
  "sub": "admin",
  "roles": ["admin"],
  "permissions": [
    "admin:*",
    "clients:read",
    "clients:write",
    "departments:read",
    "departments:write"
  ]
}
```

Standard JWT fields like `iat` and `exp` are also added automatically by `jsonwebtoken`.

## JWT Basics

JWT stands for JSON Web Token.

A JWT usually has three parts:

1. Header
2. Payload
3. Signature

They are joined with dots:

```text
header.payload.signature
```

### Header

The header describes the token type and signing algorithm.

In this project, the algorithm is:

```text
HS256
```

That means HMAC-SHA256 is used to sign the token with a shared secret.

### Payload

The payload contains claims, which are pieces of information about the user or token.

This project stores:

- `sub` for the subject or user identity
- `roles` for role names
- `permissions` for route-level authorization
- `iat` for issued-at time
- `exp` for expiration time

### Signature

The signature proves that the token was generated by someone who knows the secret.

If the payload is changed without the secret, signature verification fails and the token is rejected.

## JWT in This Project

`src/utils/token.js` is responsible for signing and verifying JWTs.

### Token generation

`generateToken(payload)` calls:

```js
jwt.sign(payload, getSecret(), {
    algorithm: "HS256",
    expiresIn: getExpiresInSeconds(),
});
```

This means:

- the secret comes from `AUTH_SECRET`
- the token is signed with `HS256`
- the expiration time comes from `AUTH_TOKEN_TTL_SECONDS`

### Token verification

`verifyToken(token)` calls:

```js
jwt.verify(token, getSecret(), {
    algorithms: ["HS256"],
});
```

This checks:

- the token was signed with the same secret
- the algorithm is allowed
- the token is not expired
- the token format is valid

If verification passes, the decoded payload is returned and assigned to `req.user`.

## Signing vs Encryption

These terms are often mixed up, so it helps to separate them clearly.

### Signing

Signing proves integrity and authenticity.

In this project, JWT uses signing:

- the server creates a signature using `AUTH_SECRET`
- the server later verifies that signature
- if the token was modified, verification fails

Signing does not hide the payload.

### Encryption

Encryption is used to hide data so only authorized parties can read it.

This project does not encrypt JWT payloads.

So if a token contains:

```json
{
  "sub": "admin",
  "roles": ["admin"]
}
```

that data can be decoded by anyone who has the token.  
They still cannot forge a valid new token without the secret, but they can read the payload.

### Hashing

Hashing is different from both signing and encryption.

Hashing is one-way:

- you can hash a value
- you cannot safely reverse the hash back to the original value

This project uses hashing for passwords through `bcrypt`.

## bcrypt Basics

`bcrypt` is used to store or compare passwords safely.

In this project, `src/services/auth.service.js` uses:

- `bcrypt.hashSync(...)`
- `bcrypt.compare(...)`

### Why bcrypt is used for passwords

Passwords should not be stored as plain text.

Instead, a password should be hashed, and later the submitted password is checked against that hash.

This helps protect users if a database or config source is exposed.

### How bcrypt works conceptually

`bcrypt`:

- hashes the password
- automatically includes a salt
- is intentionally slow compared to basic hash functions like SHA-256

That slowness is useful because it makes brute-force guessing more expensive.

### Salt

A salt is random data added before hashing.

Its purpose is to make equal passwords produce different hashes and to make precomputed attacks less effective.

With `bcrypt`, the salt is built into the hash format automatically.

## How bcrypt Is Used in This Project

In `src/services/auth.service.js`, the expected password is loaded like this:

```js
const expectedPasswordHash =
    process.env.AUTH_PASSWORD_HASH ||
    bcrypt.hashSync(process.env.AUTH_PASSWORD || "password123", 10);
```

Then the submitted password is checked with:

```js
const isPasswordMatch = await bcrypt.compare(String(password), expectedPasswordHash);
```

This supports two setups:

### Preferred setup

Use `AUTH_PASSWORD_HASH`.

That means you store a bcrypt hash in the environment instead of a plain password.

Example:

```text
AUTH_PASSWORD_HASH=$2b$10$...
```

### Development-friendly setup

Use `AUTH_PASSWORD`.

In that case, the app hashes the plain environment password at runtime and compares incoming passwords to that hash.

This is convenient for local development, but storing plain passwords in environment configuration is not ideal for production.

## Why This Is Not "Password Encryption"

It is more accurate to say:

- passwords are hashed with `bcrypt`
- JWTs are signed with `HS256`
- JWT payloads are not encrypted

Calling password storage "encryption" is common in conversation, but technically incorrect here because the password is not meant to be decrypted later.

## Security Recommendations

For real deployments, the safer approach is:

- set a strong `AUTH_SECRET`
- store `AUTH_PASSWORD_HASH` instead of `AUTH_PASSWORD`
- avoid default credentials
- keep JWT expiration reasonably short
- do not place sensitive secrets inside the JWT payload
- send tokens only over HTTPS
- rotate credentials and secrets when needed

## Quick Comparison

| Method | Used for | Reversible | Used here |
| --- | --- | --- | --- |
| Encryption | Hiding data | Yes, with a key | No |
| Hashing | Password storage/checking | No | Yes, with `bcrypt` |
| Signing | Tamper detection and authenticity | No payload hiding | Yes, with JWT `HS256` |

## Roles and Permissions

Permissions are defined in `src/auth/permissions.js`:

```js
const Permissions = Object.freeze({
    CLIENTS_READ: "clients:read",
    CLIENTS_WRITE: "clients:write",
    DEPARTMENTS_READ: "departments:read",
    DEPARTMENTS_WRITE: "departments:write",
    ADMIN: "admin:*",
});
```

Role mapping is defined in `src/auth/roleManager.js`:

```js
const ROLE_DEFINITIONS = Object.freeze({
    admin: [Permissions.ADMIN],
    manager: [Permissions.CLIENTS_READ, Permissions.CLIENTS_WRITE, Permissions.DEPARTMENTS_READ],
    viewer: [Permissions.CLIENTS_READ, Permissions.DEPARTMENTS_READ],
});
```

### Built-in roles

- `admin`
  Grants `admin:*`, and by convention expands to all currently defined permissions.
- `manager`
  Grants:
  - `clients:read`
  - `clients:write`
  - `departments:read`
- `viewer`
  Grants:
  - `clients:read`
  - `departments:read`

### Important detail about `admin`

`resolvePermissionsForRoles()` treats `admin:*` as a wildcard and returns all known permissions.  
The authorization middleware also checks for `admin:*`, so admin users automatically pass permission checks.

## Authentication Middleware

`src/middleware/auth.middleware.js` protects routes by checking:

```http
Authorization: Bearer <token>
```

If the token is valid:

- `verifyToken()` decodes it
- the decoded payload is stored on `req.user`
- the request continues

If the header is missing or the token is invalid, the API returns `401`.

Example `req.user`:

```json
{
  "sub": "admin",
  "roles": ["admin"],
  "permissions": [
    "admin:*",
    "clients:read",
    "clients:write",
    "departments:read",
    "departments:write"
  ],
  "iat": 1710000000,
  "exp": 1710003600
}
```

## Authorization Middleware

`src/middleware/authorize.middleware.js` provides four helpers:

- `requireRoles(requiredRoles)`
- `requireAnyRole(requiredRoles)`
- `requirePermissions(requiredPermissions)`
- `requireAnyPermission(requiredPermissions)`

### Behavior

- `requireRoles(["manager", "viewer"])`
  User must have both roles.
- `requireAnyRole(["manager", "viewer"])`
  User must have at least one of those roles.
- `requirePermissions(["clients:write"])`
  User must have every listed permission.
- `requireAnyPermission(["clients:write", "departments:write"])`
  User must have at least one listed permission.

If authorization fails, the API returns `403 Forbidden`.

## How Routes Use It

### Public auth routes

`src/routes/auth.routes.js` exposes:

- `POST /api/v1/auth/login`
- `GET /api/v1/auth/me`

`/me` is protected with `authenticate`, so it returns the decoded JWT payload for the current user.

### Client routes

In `src/routes/client.routes.js`:

```js
router.use(authenticate);

router.get("/", requirePermissions(Permissions.CLIENTS_READ), ClientController.getAll);
router.post("/", requirePermissions(Permissions.CLIENTS_WRITE), createValidator, ClientController.create);
```

This means:

- all client routes require a valid JWT
- read routes require `clients:read`
- create, update, and delete routes require `clients:write`

One exception is:

```js
router.get("/with-departments", ClientController.getAllWithDepartments);
```

This route is authenticated because of `router.use(authenticate)`, but it does not currently have a permission check.

### Client v2 routes

`src/routes/client.routes.v2.js` follows the same pattern.

It also contains:

```js
router.get("/test", ClientController.getTest);
```

This route currently requires authentication only, not a permission.

### Department routes

In `src/routes/department.routes.js`:

- read endpoints require `departments:read`
- create, update, and delete endpoints require `departments:write`

## Example Usage

### 1. Log in and get a token

```bash
curl -s http://localhost:3000/api/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":"password123"}'
```

Response:

```json
{
  "token": "..."
}
```

### 2. Call `/me`

```bash
curl -s http://localhost:3000/api/v1/auth/me \
  -H "Authorization: Bearer YOUR_TOKEN"
```

### 3. Access a read-protected route

```bash
curl -s http://localhost:3000/api/v1/clients \
  -H "Authorization: Bearer YOUR_TOKEN"
```

### 4. Access a write-protected route

```bash
curl -s -X POST http://localhost:3000/api/v1/clients \
  -H "Authorization: Bearer YOUR_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"name":"Alice","email":"alice@example.com"}'
```

## How To Protect a New Route

### Require authentication only

```js
router.get("/profile", authenticate, UserController.profile);
```

### Require a permission

```js
const { requirePermissions } = require("../middleware/authorize.middleware");
const { Permissions } = require("../auth/permissions");

router.post(
    "/",
    authenticate,
    requirePermissions(Permissions.CLIENTS_WRITE),
    ClientController.create
);
```

### Require any one of several permissions

```js
const { requireAnyPermission } = require("../middleware/authorize.middleware");

router.get(
    "/reports",
    authenticate,
    requireAnyPermission([
        Permissions.CLIENTS_READ,
        Permissions.DEPARTMENTS_READ,
    ]),
    ReportController.getAll
);
```

### Require a role directly

```js
const { requireRoles, requireAnyRole } = require("../middleware/authorize.middleware");

router.get("/admin-only", authenticate, requireRoles("admin"), AdminController.index);
router.get("/staff", authenticate, requireAnyRole(["admin", "manager"]), StaffController.index);
```

## How To Add a New Role

1. Add or reuse permission constants in `src/auth/permissions.js`.
2. Update `ROLE_DEFINITIONS` in `src/auth/roleManager.js`.
3. Set `AUTH_ROLES` to the desired role list.
4. Protect routes with `requirePermissions(...)`, `requireRoles(...)`, or similar middleware.

Example:

```js
const ROLE_DEFINITIONS = Object.freeze({
    admin: [Permissions.ADMIN],
    manager: [Permissions.CLIENTS_READ, Permissions.CLIENTS_WRITE, Permissions.DEPARTMENTS_READ],
    viewer: [Permissions.CLIENTS_READ, Permissions.DEPARTMENTS_READ],
    editor: [Permissions.CLIENTS_WRITE],
});
```

Then start the app with:

```bash
AUTH_ROLES=editor npm start
```

## Multiple Roles

`AUTH_ROLES` can be a comma-separated string, for example:

```bash
AUTH_ROLES=manager,viewer
```

`normalizeRoles()` converts that to:

```json
["manager", "viewer"]
```

`resolvePermissionsForRoles()` merges the permissions from all assigned roles and removes duplicates.

## Summary

This implementation is a hybrid of:

- JWT authentication for identity
- roles for easier user configuration
- permissions for route-level access control

In practice:

- login creates a JWT
- the JWT carries roles and permissions
- `authenticate` verifies the token
- authorization middleware checks access before the controller runs
